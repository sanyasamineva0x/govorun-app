# List Formatter — детекция маркеров списков в диктовке

## Контекст

Человек диктует: «нужно купить первое молоко второе хлеб третье масло». Маркеры явные — модели не нужно думать, нужно распознать маркер и поставить перенос строки + номер/дефис. Механическая задача, как удаление филлеров.

## Решение

Новый `ListFormatter` в `Core/` — deterministic final pass после gate/style, работает в обоих режимах (Standard и Super).

### Место в пайплайне

```
STT → Dictionary → DeterministicNormalizer → [LLM → Gate] → Style → ListFormatter → Insert
```

Вызывается в двух точках кода:

1. **`applyPostProcessing()`** в `PipelineEngine` — после `applyDeterministic`. Покрывает все пути, кроме standalone snippet и LLM success:
   - trivial path (строка 562) — `isTrivial` или Standard mode
   - embedded snippet, mechanical fallback без LLM (строка 436)
   - embedded snippet, LLM output или LLM failed fallback (строка 539)
   - LLM failed, plain text fallback (строка 610)

2. **`postflight()`** в `NormalizationPipeline` — после `finalText` (LLM success + gate pass/reject)

**Standalone snippets** (ветка `.standalone` в PipelineEngine) не проходят через `applyPostProcessing` вообще — возвращаются verbatim. ListFormatter их не видит. Закреплено тестом `test_standalone_snippet_preserves_lowercase_verbatim`.

### Почему final pass, а не до LLM

- Не трогает LLM/Gate — маркеры проходят как обычные слова, форматирование после финального текста
- Работает одинаково в Standard и Super
- Gate не reject'ит из-за добавленных `\n`

## Маркеры

**Все семьи требуют 2+ совпадений.** Одиночный маркер любой семьи не является списком — «первое впечатление хорошее», «во-первых, это неправда» — обычная речь, не список.

### Нумерованные семьи (выход: `1. ...`)

| Семья | Маркеры |
|-------|---------|
| Порядковые | первое, второе, третье, четвёртое, пятое, шестое, седьмое, восьмое, девятое, десятое |
| Наречия | во-первых, во-вторых, в-третьих, в-четвёртых, в-пятых |
| Пункты | пункт один, пункт два, ... пункт десять |

### Дефисные семьи (выход: `– ...`)

| Маркеры |
|---------|
| плюс, также, ещё, далее, кроме того, помимо этого, а также |

## Подход: Hybrid

- Все семьи требуют 2+ совпадений для активации
- Нумерованные семьи → `1. 2. 3.`, дефисные → `– – –`
- **Приоритет по силе сигнала, не по позиции:** сканируем все семьи, выбираем ту, у которой 2+ совпадений. Если несколько семей набрали порог — нумерованные (порядковые/наречия/пункты) побеждают дефисные

## Выходной формат

Маркер удаляется, заменяется на номер или дефис.

### Капитализация

Зависит от стиля. API: `format(_ text: String, style: SuperTextStyle?) -> String`.
- `relaxed` — без капитализации (стиль держит строчные)
- `normal`, `formal`, `nil` — capitalize первую букву каждого пункта

### Очистка пунктуации

После удаления маркера выполняется boundary cleanup:
- Strip ведущих запятых, тире (—, –, -), двоеточий у каждого item
- Strip trailing пунктуации (точки, запятые, точки с запятой) у каждого item
- Trim whitespace

Пункты списка не содержат завершающей пунктуации — ни точек, ни запятых, ничего.

**Правила пунктуации для списков:**
- Завершающая пунктуация у пунктов запрещена во всех стилях
- formal-правило terminal period применяется только к plain text, НО НЕ к list items

### Шапка

Текст до первого маркера. **Двоеточие не добавляется автоматически** — если пользователь продиктовал двоеточие, оно уже есть. Шапка просто получает `\n` перед списком.

### Примеры

```
нужно купить первое молоко второе хлеб третье масло
→ нужно купить
1. Молоко
2. Хлеб
3. Масло
```

```
плюсы этого подхода во-первых скорость во-вторых простота в-третьих цена
→ плюсы этого подхода
1. Скорость
2. Простота
3. Цена
```

```
во-первых, скорость, во-вторых, простота.
→ 1. Скорость
2. Простота
(запятые после маркеров и trailing точка удалены)
```

```
плюс скорость плюс простота плюс цена
→ – Скорость
– Простота
– Цена
```

```
первое впечатление хорошее
→ первое впечатление хорошее  (одиночный маркер — без изменений)
```

```
плюс этого подхода в скорости
→ плюс этого подхода в скорости  (одиночный маркер — без изменений)
```

```
ещё одна вещь во-первых скорость во-вторых простота
→ ещё одна вещь
1. Скорость
2. Простота
(«ещё» — одиночный дефисный, игнорируется; «во-первых/во-вторых» — нумерованная семья с 2+ побеждает)
```

```
(relaxed mode) нужно купить первое молоко второе хлеб
→ нужно купить
1. молоко
2. хлеб
```

## Edge cases

- **Шапка** — текст до первого маркера, без автоматического двоеточия
- **Маркер в начале** — список без шапки
- **Текст после последнего пункта** — часть последнего пункта (trailing пунктуация всё равно strip'ится)
- **Приоритет семей** — нумерованные побеждают дефисные при конкуренции; одиночные маркеры из проигравшей семьи остаются как обычный текст
- **Standalone snippets** — не трогать, verbatim контракт
- **Пунктуация вокруг маркеров** — запятые, тире, двоеточия после маркера удаляются; trailing пунктуация у каждого item удаляется
- **Relaxed style** — пункты без капитализации

## Архитектура

```swift
// Core/ListFormatter.swift
enum ListFormatter {
    static func format(_ text: String, style: SuperTextStyle? = nil) -> String
}
```

Без состояния, без протоколов — чистая функция. Style нужен только для решения о капитализации.

## Тесты

`GovorunTests/ListFormatterTests.swift`:
- Порядковые 2+ → нумерованный список
- Наречия 2+ → нумерованный список
- Пункты 2+ → нумерованный список
- Дефисные 2+ → дефисный список
- Одиночный порядковый → без изменений
- Одиночный наречие → без изменений
- Одиночный дефисный → без изменений
- Шапка без автоматического двоеточия
- Шапка с продиктованным двоеточием — сохраняется
- Маркер в начале (без шапки)
- Конкуренция семей: нумерованная побеждает дефисную
- Слабая семья не блокирует сильную (ранний одиночный «ещё» + поздние «во-первых/во-вторых»)
- Капитализация первой буквы в normal/formal
- Без капитализации в relaxed
- Cleanup запятых/тире после маркера («во-первых, скорость» → «Скорость»)
- Strip trailing пунктуации у items (точки, запятые, точки с запятой)
- Без маркеров — без изменений
- Пустая строка — без изменений

Интеграционные:
- `NormalizationPipelineTests` — postflight прогоняет через ListFormatter
- `PipelineEngineTests` — trivial path форматирует, standalone snippet — нет
