# LLM-нормализация: дорожная карта

Стратегический план развития Говоруна от базовой очистки текста до голосового AI-агента.

> **Структура документа:** Уровни 0-2 — MVP track (основной). Уровни 3-8 — Future Bets (отдельный горизонт планирования, реализация после валидации MVP).

## Текущее состояние

```
GigaAM-v3 e2e_rnnt (~200ms, локально)
    ↓
DeterministicNormalizer (~1ms, regex: числа, даты, филлеры)
    ↓
[Говорун Super] GigaChat3.1 Q4_K_M через llama-server (1-3s)
    ↓                     ↑
NormalizationGate         static arm64 binary в Contents/Helpers/
    ↓
TextInserter (AX API / clipboard)
```

Уровни 0-1 реализованы. Static bundling llama-server завершён (PR #16).
Deterministic pipeline работает всегда. LLM — opt-in (Говорун Super).

## Целевой pipeline (MVP)

```
GigaAM-v3 (~200ms)
    ↓
DeterministicNormalizer (~1ms) — числа, даты, время (детерминированно)
    ↓
GigaChat3.1-10B-A1.8B Q4_K_M — филлеры, self-correction, стиль
    ↓
Normalization Gate — контракт зависит от режима (см. Safety Gate)
    ↓
TextInserter
```

Целевая латентность: **p95 < 800ms** на 16 GB Mac. Подлежит валидации на уровне 0.

## Выбор модели: GigaChat3.1-10B-A1.8B

### Почему именно она

| Параметр | Значение |
|----------|----------|
| Архитектура | MoE (Mixture of Experts) |
| Параметров всего | 10B |
| Активных при инференсе | 1.8B |
| Лицензия | разрешительная |
| Контекст | 32K (Lightning — 256K) |
| Русский (MMLU_RU) | 0.68 (лучше Qwen3-4B) |
| Function calling | Есть |
| Формат | GGUF (Q4_K_M = 6.5 GB) |

### Почему не FRED-T5

FRED-T5 — encoder-decoder, без fine-tune бесполезен для нормализации. GigaChat3.1 — instruct-модель, работает из коробки с промптом. Fine-tune можно добавить позже, но MVP не блокируется.

### Почему не Qwen/Llama

GigaChat3.1 обучен на 20T токенов с фокусом на русский. MMLU_RU 0.68 vs Qwen3-1.7B 0.49 — значительное преимущество для задачи нормализации русской речи. Плюс разрешительная лицензия и function calling из коробки.

### Требования к RAM (подлежат валидации на уровне 0)

```
GigaAM:         ~900 MB
GigaChat3.1 Q4: ~7-8 GB (модель + KV-cache)  ← ТРЕБУЕТ ЗАМЕРА
macOS:          ~3-4 GB
────────────────────────
Итого:          ~12-13 GB → минимум 16 GB RAM (гипотеза)
```

Для 8 GB маков — дистилляция в Qwen3-0.6B (~400 MB), см. раздел «Дистилляция».

## MVP Track (Уровни 0-2)

### Уровень 0: Benchmark & Feasibility ✅
**Завершён (PR #13, 2026-03-26)**

#### Что делаем

1. Скачать GigaChat3.1-10B-A1.8B Q4_K_M GGUF
2. Поднять через llama.cpp / llama-cpp-python
3. Прогнать **100 тестовых фраз** (короткие 3-5 слов, средние 10-20, длинные 30-50)
4. Замерить на **реальном железе** (M1 8GB, M1 16GB, M2 16GB, M3 24GB — что доступно)

#### Acceptance criteria

| Метрика | Порог (go) | Порог (пересмотр) |
|---------|-----------|-------------------|
| p50 latency (полный ответ) | < 500ms | > 800ms |
| p95 latency (полный ответ) | < 800ms | > 1500ms |
| First token latency | < 100ms | > 300ms |
| Peak RAM (модель + KV-cache) | < 8 GB | > 10 GB |
| GigaAM + GigaChat3.1 одновременно | Работает на 16 GB | OOM на 16 GB |
| Качество нормализации (ручная оценка) | 80%+ фраз корректны | < 60% корректны |

#### Решения по результатам

- **Всё green** → переход на уровень 1
- **Latency red** → рассмотреть GigaChat3.1 Lightning, Qwen3-0.6B, или SAGE (FRED-T5)
- **RAM red на 16 GB** → последовательная загрузка (GigaAM → выгрузить → GigaChat3.1), или дистилляция
- **Качество red** → пересмотр промпта, рассмотреть fine-tune или SAGE

### Уровень 1: Базовая очистка ✅
**Завершён (PR #14, 2026-03-27)**

System prompt v3b (97% quality на 36 samples). Реализовано:
- Удаление филлеров: «ну», «типа», «как бы», «вот», «это самое»
- Self-correction: «встреча в среду, нет, в четверг» → «встреча в четверг»
- Пунктуация и капитализация поверх GigaAM e2e
- Нормализация: «двадцать третье марта» → «23 марта 2026»

```
Вход:  "ну вот значит 15 нет 20 тысяч рублей перевести"
Выход: "Перевести 20 000 ₽."
```

**Контракт:** string in → string out. Normalization gate.

**Инфраструктура (PR #15, #16):**
- SuperAssetsManager: discovery llama-server + GGUF, state machine
- Static llama-server arm64 bundling (cmake, 12M, Contents/Helpers/)
- CI: cache + build + validate (arch + allowlist deps)
- Symlink resolution, race condition fixes в UI

### Уровень 2: Стили текста (Super-only)
**Срок: 1 неделя | Сложность: низкая**

Три стиля по уровню формальности (только для Говорун Super):

| Стиль | Назначение | Пример выхода |
|-------|-----------|---------------|
| Расслабленный | Мессенджеры, личные чаты | строчные, без точек, бренды кириллицей (слак, зум) |
| Обычный | Заметки, браузеры, IDE | С заглавной, без точки, бренды оригиналом |
| Формальный | Почта, деловая переписка | С заглавной, точка, сленг раскрыт, бренды оригиналом |

Пример:
```
Вход:  "ну скинь в слак что митинг в четверг"
Расслабленный: "скинь в слак что митинг в четверг"
Обычный:       "Скинь в Slack, что митинг в четверг"
Формальный:    "Отправьте в Slack, что совещание в четверг."
                                                    ↑ rewriting — Level 2.5
```

Переключение: авто (bundleId mapping) или ручной (один стиль на всё). Отдельный `SuperTextStyle` enum, classic TextMode не затрагивается.

**Контракт:** все три стиля → normalization gate. Rewriting (формальный с переписыванием) вынесен в уровень 2.5.

### Уровень 2.5: Rewriting contract для формального стиля
**Срок: 1 неделя | Сложность: средняя**

Формальный стиль переключается с normalization на rewriting contract:
- Морфологическое ты → Вы (спряжение глаголов)
- Переписывание разговорных оборотов в деловой стиль
- Rewriting Gate вместо Normalization Gate (NER + длина, без edit distance)

Требует: отдельный seed corpus для оценки качества переписывания.

**Полный design spec:** `docs/superpowers/specs/2026-03-29-text-styles-v2-design.md`

## Future Bets (Уровни 3-8)

> Реализация после валидации MVP (уровни 0-2). Каждый уровень требует отдельного design spec перед началом работы.

### Уровень 3: Sound Like You (персонализация)
**Срок: 2-3 недели | Сложность: средняя**

> ⚠️ **Требует Privacy Spec** (см. раздел «Privacy Model») до начала реализации.

Приложение следит: пользователь надиктовал → получил текст → вручную поправил. Каждая пара «до/после» сохраняется локально (SwiftData).

Через 50-100 пар — профиль стиля. Последние 10-20 примеров вставляются в промпт как few-shot:

```
Системный промпт:
"Пользователь обычно пишет так:
- «привет всем» → «Привет!»
- «ну давайте обсудим» → «Обсудим.»
- «короче я думаю что» → «Считаю, что»
Перепиши транскрипт в его стиле."
```

GigaChat3.1 Lightning с 256K контекстом может держать сотни примеров. Модель копирует привычки: предпочтение тире или запятых, длину предложений, обращения, характерные обороты.

Инфраструктура частично есть — PostInsertionMonitor уже отслеживает правки.

**Дополнение: автоопределение порога.** Метрика: «модель предсказывает правки с точностью 80%+ → персонализация готова». UI: «Говорун изучил ваш стиль на 73%».

### Уровень 4: Контекстная осведомлённость
**Срок: 1-2 недели | Сложность: средняя**

> ⚠️ **Требует Privacy Spec** (см. раздел «Privacy Model») до начала реализации.

Через Accessibility API приложение видит контекст:
- Активное приложение и заголовок окна
- Имя файла (в IDE)
- Последние сообщения в чате (через AX)
- Тип поля ввода

GigaChat3.1 получает контекст в system prompt и адаптируется автоматически:
- В мессенджере → короткие неформальные сообщения
- В почтовом клиенте → деловой стиль с приветствием
- В IDE → код-комментарии, форматирование переменных (camelCase/snake_case по файлу)

**Уже реализовано частично:**
- `AppContextEngine` — определяет активное приложение
- `appName` — передаётся в pipeline
- `textMode` — режим текста
- `AppModeSettingsView` — настройки режимов по приложениям

Нужно расширить: не только appName, но заголовок окна, имя файла, выделенный текст. Передавать в system prompt.

**Дополнение: контекст переписки.** Через AX API читать последние 2-3 сообщения → модель видит тон беседы и подстраивается. «Привет, как дела?» → неформальный ответ. «Уважаемый Алексей Владимирович» → формальный.

AquaVoice называет это «Deep Context» и продаёт как premium-фичу.

### Уровень 5: Голосовые команды
**Срок: 3-4 недели | Сложность: высокая**

> ⚠️ **Архитектурный разрыв:** требует перехода от string-контракта к structured output. См. раздел «Архитектурные контракты».

Не просто диктовка, а команды смешанные с текстом:

```
"Напиши Иванову что встреча переносится на четверг. Точка.
 Новый абзац. Добавь что нужно подготовить отчёт по проекту Алтай."
```

GigaChat3.1 парсит: что текст, что команда.
- «Точка» и «новый абзац» → форматирование
- «Напиши Иванову» → метаинформация
- Остальное → контент

Сложные команды: «Выдели последнее предложение жирным», «Замени Иванов на Петров», «Удали предыдущий абзац». Модель генерирует не только текст, но и структурированные команды через function calling.

Набор функций: `insert_text`, `format_bold`, `delete_last`, `replace`, `new_paragraph`.

**Ограничения:**
- Смешанный ввод (текст + команды в одном потоке) — сложная задача парсинга
- AX API — не все приложения поддерживают программное форматирование
- Нужен undo для деструктивных команд

**Дополнение: «Подтверди перед выполнением»** для деструктивных команд. «Удали всё» → bottom bar показывает «Удалить весь текст?» с кнопками Да/Нет.

### Уровень 6: Email/документ из голоса
**Срок: 1-2 недели | Сложность: средняя**

Пользователь бубнит 2 минуты в поток сознания → GigaChat3.1 структурирует:

```
Промпт: "Структурируй как деловое письмо: приветствие, суть,
         список действий, подпись."
→ Готовое письмо с bullet points, нумерацией, подписью.

Промпт: "Оформи как протокол совещания: дата, участники,
         обсуждённые вопросы, решения, ответственные."
→ Структурированный документ из 20-минутной записи.
```

GigaChat3.1 с 32K контекстом обработает ~3000 слов (20 минут речи).

Нужно:
- Режим «длинная запись» — не вставлять сразу, а показать preview
- Шаблоны: письмо, протокол, отчёт, ТЗ
- UI для редактирования перед вставкой

**Дополнения:**
- «Дополни» — надиктовал начало вчера, сегодня «добавь что дедлайн сдвигается» → модель дописывает в том же стиле
- Экспорт: «Скопировать» / «Открыть в Pages» / «Отправить на почту»

### Уровень 7: Мультимодальный пайплайн (агент)
**Срок: 2-3 месяца | Сложность: высокая**

GigaAM-v3 (транскрибация) → GigaChat3.1 (понимание + function calling) → действие:

```
"Создай задачу в Трекере: починить баг с авторизацией,
 приоритет высокий, дедлайн пятница"
→ GigaChat3.1 парсит в JSON → API-вызов в Яндекс.Трекер/Jira

"Найди в почте последнее письмо от Петрова и ответь что я согласен"
→ GigaChat3.1 формулирует ответ → API-вызов
```

Это уже голосовой агент, не диктовка. Но инфраструктура та же.

**MCP (Model Context Protocol)** — ключ к масштабированию. Вместо хардкода интеграций — стандартный протокол. Пользователь подключает MCP-сервер для Jira, другой для почты. Говорун — хаб. Claude Code, Cursor, Windsurf — все на MCP.

**Ограничение:** это уже не офлайн — вызовы к API требуют сети.

### Уровень 8: TrueConf AI Server
**Срок: 3-6 месяцев | Сложность: высокая**

Замена текущего стека AI Server:
- GigaAM вместо Whisper для транскрибации
- GigaChat3.1 вместо GLM-4 для суммаризации

Новые фичи:
- Real-time субтитры (GigaAM e2e_ctc)
- Автопротоколирование совещания
- Выделение action items («Иванов должен подготовить отчёт к пятнице» → задача)
- Поиск по архиву записей (семантический, через эмбеддинги)

Весь стек российский, с контролируемым лицензированием, реестр ПО закрыт.

**Дополнения:**
- Emotion detection (Aniemore) — «участник раздражён» в протоколе
- Speaker diarization — кто что сказал
- Автоматические теги: «решение», «вопрос», «action item», «дедлайн»

## Архитектурные контракты

### Три класса задач

Roadmap содержит задачи разной природы, которые не могут жить в одном pipeline-контракте:

| Класс | Уровни | Вход → Выход | Пример |
|-------|--------|-------------|--------|
| **Normalization** | 1, 2 (все три стиля) | string → string (близкий к входу) | Убрать филлеры, пунктуация, стиль |
| **Rewriting** | 2.5 (формальный), 3-4, 6 | string → string (может быть далёким от входа) | Деловой стиль, перевод, документ |
| **Structured Actions** | 5, 7 | string → structured output (JSON/commands) | Function calling, голосовые команды |

### Текущий контракт LLM (уровни 1-2)

```swift
protocol LLMClient {
    func normalize(_ text: String, mode: TextMode, hints: NormalizationHints) async throws -> String
}
```

Достаточен для MVP (уровни 0-2).

### Будущий контракт (уровни 5+)

```swift
enum ProcessingResult {
    case text(String)
    case command(VoiceCommand)
    case document(DocumentDraft)
    case mixed([ProcessingResult])
}

protocol LLMProcessor {
    func process(_ text: String, context: AppContext) async throws -> ProcessingResult
}
```

**Решение:** не проектировать ProcessingResult сейчас. Реализовать `LLMClient.normalize()` для MVP, расширить до `LLMProcessor.process()` когда дойдём до уровня 5. Преждевременная абстракция хуже, чем рефакторинг по месту.

## Safety Gate

### Три контракта валидации

Один gate на все режимы не работает. Каждый класс задач — свой validator и свой fallback:

#### 1. Normalization Gate (уровни 1-2: «как сказал», «чисто»)

| Проверка | Порог | Что ловит |
|----------|-------|-----------|
| Edit distance | ≤40% токенов (≤25% если < 10 токенов) | Галлюцинации |
| Frozen NER entities | Все сохранены | Фактические ошибки |
| Semantic similarity | ≥0.85 (rubert-tiny2) | Смысловые искажения |

Fallback: выход DeterministicNormalizer (без LLM).

#### 2. Rewriting Gate (уровень 2: «формально», перевод)

| Проверка | Порог | Что ловит |
|----------|-------|-----------|
| Frozen NER entities | Все сохранены | Потеря фактов при переписывании |
| Длина результата | ±50% от входа | Обрезка или раздувание |
| Не пустой результат | len > 0 | Модель вернула пустоту |

Semantic similarity **не используется** — при переписывании в формальный стиль или переводе similarity может быть низкой, и это нормально. Edit distance **не используется** — формальный стиль может полностью переписать фразу.

Fallback: выход Normalization Gate (режим «чисто»).

#### 3. Action Gate (уровень 5+: голосовые команды)

| Проверка | Порог | Что ловит |
|----------|-------|-----------|
| JSON schema valid | Strict | Невалидный structured output |
| Команда в allowlist | Whitelist | Неизвестные/опасные команды |
| Деструктивные действия | Подтверждение пользователя | Случайное удаление |

Fallback: вставить текст as-is (как будто команды нет).

**Принцип: лучше грязно, чем неправильно.**

## Privacy Model

> Обязательно к реализации **до** уровней 3-4. Без privacy spec эти уровни нельзя начинать.

### Уровень 3 (персонализация): хранение пар «до/после»

| Аспект | Решение |
|--------|---------|
| Opt-in | По умолчанию **выключено**. Включается явно в настройках |
| Retention | Максимум 90 дней, потом автоудаление |
| Шифрование | Пары шифруются AES-256-GCM, ключ в Keychain |
| Удаление | Кнопка «Удалить все данные обучения» в настройках |
| UX | Тоггл «Учиться на моих правках» + описание что хранится |
| Исключения | Не сохранять пары из приложений в denylist |

### Уровень 4 (контекст): чтение содержимого окна

| Аспект | Решение |
|--------|---------|
| Opt-in | По умолчанию **выключено**. Включается явно в настройках |
| Per-app denylist | Список приложений, из которых **никогда** не читать контекст (1Password, банки, Keychain Access, терминал) |
| Глубина | Только заголовок окна + имя файла (по умолчанию). Чтение сообщений — отдельный opt-in |
| Индикатор | Иконка в bottom bar когда контекст активен |
| Данные не покидают устройство | Контекст передаётся в промпт, не сохраняется на диск |
| Sensitive content | Не передавать в промпт поля с type=password |

### Общие принципы

- Все данные персонализации — **только на устройстве**, никогда не покидают Mac
- Пользователь может **в любой момент** удалить все собранные данные
- **Uninstall не гарантирует очистку.** На macOS удаление .app не трогает `~/Library/Application Support/`, SwiftData, Keychain. Необходимо: (1) кнопка «Удалить все данные» в настройках (явный purge), (2) документация для пользователя о том, где хранятся данные
- Denylist по умолчанию: менеджеры паролей, банковские приложения, терминалы, VPN-клиенты

## Дополнительные фичи (вне уровней)

### Перевод (2.5)
Надиктовал по-русски → текст по-английски (или наоборот). GigaChat3.1 поддерживает 10 языков. Один промпт. Контракт: rewriting gate (NER + длина). Вынесено из MVP, чтобы не раздувать основную ставку «лучший русский voice typing». Отдельный продуктовый режим с собственной оценкой качества, UX-ожиданиями и failure cases.

### Мультиязычность (2.5)
Пользователь говорит на русском с английскими вставками. GigaChat3.1 оставляет как есть: «Запусти docker compose up на сервере» → не переводит docker compose.

### Корпоративный словарь (3.5)
DictionaryStore расширить: импорт/экспорт CSV, шаринг между коллегами, специализированные пакеты (медицина, право, IT).

### Timestamps (4.5)
GigaAM возвращает таймкоды слов. В длинных записях (уровень 6) — клик на слово → слышишь этот момент. Частично есть в HistoryView.

### Обратная связь (9)
Голосовой ввод → текст → приложение отвечает → TTS (Silero) → пользователь слышит ответ. Полностью hands-free цикл. Для accessibility — killer feature.

## Дистилляция для 8 GB маков

### Проблема
GigaChat3.1 Q4_K_M (6.5 GB) + GigaAM (900 MB) + macOS (3-4 GB) = ~12 GB. На 8 GB маке не влезет. **Точные цифры — после уровня 0.**

### Решение: дистилляция в Qwen3-0.6B

```
Teacher: GigaChat3.1-10B (6.5 GB, умная)
    ↓ генерирует 100K пар «грязный → чистый»
Student: Qwen3-0.6B (400 MB, быстрая)
    ↓ учится повторять
```

| Шаг | Что | Время | Стоимость |
|-----|-----|-------|-----------|
| 1 | Генерация 100K пар через teacher | 2-3 дня (Mac) | $0 |
| 2 | Fine-tune student (LoRA rank 16) | 2-4 часа (A100) | $5-10 |
| 3 | Конвертация в GGUF Q4_K_M | 5 минут | $0 |

Ожидаемый результат: **400 MB модель, быстрый инференс**. Точные цифры качества и латентности — после эксперимента. Заявка «90-95% качества teacher'а» — гипотеза, не гарантия.

### Двухуровневая стратегия

```
Mac 8 GB:   GigaAM → regex → Qwen3-0.6B distilled (400 MB) → gate
Mac 16 GB+: GigaAM → regex → GigaChat3.1 Q4 (6.5 GB) → gate
```

Приложение определяет RAM при старте и загружает подходящую модель.

## Flywheel (самоулучшение)

> Зависит от Privacy Spec (уровень 3). Данные для flywheel — те же пары «до/после».

```
Пользователь диктует → получает текст → правит вручную
    ↓
Пара «что выдала система → что хотел пользователь» → локальный датасет
    ↓
Накопили 10K пар → дообучили модель (LoRA) → лучше → меньше правок → repeat
```

Каждый пользователь бесплатно размечает данные. Через 6 месяцев — модель заточена под реальную речь. Невоспроизводимое преимущество.

## Ключевой инсайт

GigaChat3.1 при 1.8B активных параметрах — не «тупая маленькая модель». Это модель уровня 4-8B dense, которая быстро работает. Она может быть **мозгом продукта**, а не просто постпроцессором.

Разница между «диктовка с очисткой» и «голосовой интерфейс к компьютеру» — ровно один слой LLM-обработки. GigaChat3.1 достаточно умна для этого слоя.

**Но прежде чем строить на этом предположении — уровень 0 должен его подтвердить.**
